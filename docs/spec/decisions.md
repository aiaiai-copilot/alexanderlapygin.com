# Журнал решений (decisions.md)

Документ фиксирует **архитектурные и продуктовые решения** проекта alexanderlapygin.com с обоснованиями и отвергнутыми альтернативами. Это сопровождение к [`spec.md`](./spec.md) — само ТЗ описывает *что* система делает, этот документ — *почему именно так*.

Формат каждой записи: контекст → решение → отвергнутые альтернативы → последствия.

## 1. Архитектурные решения

### 1.1. Статика по умолчанию, SSR только для контактной формы

**Контекст.** Сайт — преимущественно read-only витрина с одной интерактивной точкой (форма обращения). Большинство страниц могут быть полностью статичными.

**Решение.** Astro в режиме static-by-default с включённым SSR-адаптером (Node) только для маршрута `/api/contact`. Все страницы пререндерятся на сборке.

**Альтернативы.** Полностью SPA (React/Next с SSR на каждый запрос) — оверкилл для read-only сайта. Ставит Node-процесс в критический путь всех страниц: в нашей схеме падение SSR оставляет сайт работающим (статику отдаёт nginx, недоступна только форма), в полном SSR падение Node означает падение всего сайта. Дополнительно: каждая страница рендерится при каждом запросе — нагрузка на CPU и память выше; деплой требует graceful reload или blue-green, чтобы не терять запросы; surface area для атак больше. Полностью статичная (без SSR) с внешним сервисом для формы (Formspree, Web3Forms) — добавляет трансграничную передачу, делает невозможным гибкое логирование и сохранение в `failed/`.

**Последствия.** Минимальный JS на клиенте; высокая производительность; единственный серверный процесс — слабая точка отказа, но его падение не ломает сайт (только форму). При падении SSR пользователь видит работающий сайт с `mailto:`-ссылкой в подвале как fallback.

### 1.2. systemd как процесс-менеджер

**Контекст.** SSR-процесс должен запускаться при ребуте, перезапускаться при падении, иметь лимиты ресурсов и интеграцию с системным логированием.

**Решение.** systemd unit `Type=simple` с `Restart=on-failure`, `MemoryMax=256M`, `StartLimitBurst=5`.

**Альтернативы.** PM2 — добавляет npm-зависимость с непрозрачной поверхностью CVE; cluster mode не нужен для одного инстанса; своё логирование вместо journalctl — лишний шаг. Docker / docker-compose — оверкилл для одного процесса на одной VM, добавляет слой образа и оркестрации. supervisord — устаревающий, systemd покрывает все его сценарии нативно.

**Последствия.** Нет дополнительных зависимостей. Логи через `journalctl`. Падение SSR-процесса автоматически восстанавливается. Лимиты памяти защищают от утечек.

### 1.3. Стандартный `systemctl restart` без graceful reload

**Контекст.** При деплое SSR-процесс рестартует. Возможны два режима: graceful (через `sd-notify` + `Type=notify` + обработчик SIGHUP с drain) или простой `systemctl restart`.

**Решение.** `systemctl restart` без graceful reload. Простой в момент рестарта — секунды.

**Альтернативы.** `Type=notify` + `sd-notify` + watchdog + SIGHUP с drain — это даёт даунтайм в миллисекунды вместо секунд, но добавляет: npm-зависимость `sd-notify`, ~30 строк инфраструктурного кода в SSR, unit-тесты на корректность сигналов, потенциальные баги в обработчике SIGHUP. Blue-green деплой через nginx upstream — ещё сложнее, фиксирован в [`spec.md`](./spec.md) §13 как отложенное расширение.

**Триггер пересмотра.** Если регулярные «секундные» простои при `systemctl restart` начнут создавать видимые ошибки в логе (потерянные запросы при деплое, что при текущей частоте обращений и деплоев — практически невозможно).

**Последствия.** Возможна потеря отдельного запроса формы, попавшего в момент рестарта. Вероятность пересечения деплоя и обращения — крайне мала на ожидаемой нагрузке. UX-следствие: пользователь увидит ошибку и предложение написать на email напрямую.

### 1.4. certbot как стандартный путь для HTTPS

**Контекст.** Для VPS нужен HTTPS-сертификат с автопродлением, независимо от выбранного провайдера. Возможные пути: Let's Encrypt через certbot (HTTP-01 или DNS-01), или managed certificate service провайдера (Yandex Cloud Certificate Manager, Selectel SSL и подобные).

**Решение.** certbot как универсальный путь. Метод (HTTP-01 или DNS-01) выбирается в [`runbook.md`](./runbook.md) при первичной настройке.

**Альтернативы.** Managed certificate services провайдеров — обычно интегрированы только с управляемыми сервисами этого провайдера (load balancers, API gateways, object storages) и не предоставляют штатного механизма доставки приватного ключа на пользовательскую Compute VM. На Compute VPS пришлось бы выгружать ключ вручную, что снимает основной бенефит managed-сервиса.

**Последствия.** Стандартное решение, переносимое между провайдерами. Автопродление через certbot timer. HTTP-01 проще (один порт 80, никаких плагинов), DNS-01 требуется только для wildcard или закрытого 80-го порта.

### 1.5. Английская локаль с первого дня инфраструктурно

**Контекст.** Основная аудитория — русскоязычная. Англоязычная — вторичная, по факту неподтверждённая. Возможны два пути: построить двуязычную инфраструктуру сразу или начать только с RU.

**Решение.** Двуязычная инфраструктура с первого дня (маршруты `/en/`, словари UI, переключатели, hreflang, x-default). Контент в EN — частичный, наполняется по триггеру (§13.3 в [`spec.md`](./spec.md)).

**Альтернативы.** Только RU на старте, EN добавляется позже целиком. Это упростило бы инфраструктуру (один набор маршрутов, нет переключателей, нет hreflang/x-default, одна 404, одна ценовая сетка), но при появлении триггера потребовало бы редизайна URL-структуры, что болезненно для SEO существующих RU-страниц.

**Последствия.** Постоянный overhead двуязычности даже при пустой EN-локали. Зато добавление EN-контента — точечная редактура Markdown-файлов, без структурных изменений. При выборе сценария «никогда не понадобится EN» — overhead был построен впустую; вероятность этого сценария оценивается автором как невысокая, но реальная. Это сознательный риск.

### 1.6. Выбор хостинг-провайдера — отдельное решение

**Контекст.** На каком VPS-провайдере разворачивается проект. Решение влияет на: применимость ст. 12 152-ФЗ (трансграничная передача ПДн при зарубежном хостинге), стоимость, доступность DNS-плагинов для certbot, гео-латентность для целевой аудитории.

**Решение.** Конкретный провайдер **не зафиксирован в** [`spec.md`](./spec.md). Выбор делается на этапе первичной настройки и фиксируется в [`runbook.md`](./runbook.md) §2.1.

**Кандидаты для российской аудитории (без трансграничной передачи ПДн):** Yandex Cloud Compute Cloud, Selectel, VKCloud (Mail.ru Cloud Solutions), Timeweb Cloud, Beget VPS, Cloud.ru.

**Кандидаты для международной аудитории (с трансграничной передачей ПДн):** Hetzner (Германия) — оптимальный по цене/качеству для EU; DigitalOcean — широкая география; AWS Lightsail, Vultr — глобально.

**Последствия зарубежного выбора:**
- Активируется раздел 10.4 spec.md в полном виде.
- Требуется уведомление Роскомнадзора по ст. 12 152-ФЗ до начала эксплуатации.
- В политику конфиденциальности добавляется конкретный получатель и страна хостинга.
- Гео-латентность для российской аудитории выше.

**Сходство всех провайдеров.** Архитектура (Ubuntu LTS, nginx, systemd, certbot) идентична. Перенос между провайдерами — пересоздание VM, копирование секретов из 1Password, изменение DNS A-записи на новый IP. Время полной миграции — несколько часов.

**Замечание про исходящий TCP/25.** Практически все облачные провайдеры (Yandex Cloud, Hetzner, DigitalOcean, AWS, Google Cloud и т. д.) блокируют исходящий TCP/25 по умолчанию. Это не влияет на работу системы — доставка идёт через SMTP relay на порте 465 (SSL) или 587 (STARTTLS).

## 2. Продуктовые компромиссы

### 2.1. Один канал доставки обращений (email), без Telegram

**Контекст.** Возможные каналы доставки обращений автору: email через SMTP, Telegram через Bot API, иные мессенджеры.

**Решение.** Только email через SMTP relay-провайдера. Telegram не используется.

**Альтернативы.** Telegram как дополнительный канал — даёт оперативность уведомления. Но добавляет: отдельный чекбокс согласия на трансграничную передачу (Telegram Messenger Inc. расположен вне РФ), уведомление в Роскомнадзор по ст. 12 152-ФЗ, запись в политике конфиденциальности, отдельный код вызова Bot API, отдельное экранирование MarkdownV2/HTML, отдельную процедуру удаления сообщений при запросе субъекта.

**Триггер пересмотра.** Если фактическая задержка email-уведомлений станет проблемой (например, переход в режим оперативной поддержки клиентов). См. §13.6 в [`spec.md`](./spec.md).

**Последствия.** Раздел 10.4 (трансграничная передача) в политике конфиденциальности применяется только при выборе зарубежного SMTP. При российском SMTP — раздел не нужен, уведомление в РКН по ст. 12 — не требуется. Один путь экранирования, одна процедура удаления, одно согласие. Цена: задержка реакции автора — типично минуты в часы рабочего дня, что соответствует обещанию «ответ в течение суток» с большим запасом.

### 2.2. Прямые контакты только в подвале и на `/contact`

**Контекст.** Контактные ссылки (email, GitHub) могут размещаться в шапке, в тексте `/about`, в подвале, на специальной странице `/contact`. Чем больше точек контакта — тем выше шанс «прямого обращения», но тем менее типизирована воронка.

**Решение.** Контакты — только в двух местах: сквозной подвал (с иконками email и GitHub) и страница `/contact` (форма + `mailto:`). В шапке контактных иконок нет. В тексте `/about` нет `mailto:`-ссылок и аналогов.

**Альтернативы.** Прямые `mailto:`-ссылки в тексте `/about` — повышают конверсию для тех, кто уже готов написать, но снижают долю обращений через форму, которая даёт автору минимальный полезный контекст до начала диалога (тип проекта, бюджет, детали). При прямой почте — переписка часто начинается с «привет, можете сделать X?» без подробностей.

**Последствия.** Воронка направляет посетителей в форму, при этом не блокирует email (он виден в подвале и на `/contact`). Изменение контактов — точечное в одном месте, без правок текста разделов.

### 2.3. Mobile-first как основной профиль качества

**Контекст.** Качество отображения проверяется на нескольких диапазонах ширины. Можно оптимизировать в первую очередь под десктоп или под мобильные.

**Решение.** Mobile-first. Lighthouse mobile — основной профиль приёмки. Десктоп — производный.

**Обоснование.** Google индексирует сайты в mobile-режиме с 2019 года. Mobile-профиль Lighthouse существенно строже десктопного (эмулируется класс CPU и канал). Значительная доля целевой аудитории приходит с мобильных устройств. Если страница работает хорошо на mobile — на desktop она почти гарантированно работает отлично; обратное неверно.

**Последствия.** Дизайн-решения принимаются от узких экранов вверх. На странице поста допускается чуть более низкий Performance ≥ 75 vs типовые ≥ 80 — сознательный компромисс качества и функциональности (ToC, подсветка синтаксиса, OG-генерация).

### 2.4. Cookie-баннер без 12-месячного истечения согласия

**Контекст.** Cookie-баннер при первом посещении сохраняет факт согласия в `localStorage`. Можно автоматически показывать его повторно по истечении срока (например, 12 месяцев).

**Решение.** Однократный показ баннера. Без автоматического повторного показа. Пользователь может пересмотреть выбор через ссылку «Настройки cookie» в подвале.

**Альтернативы.** Повторный показ через 12 месяцев — даёт «свежее согласие», но в 152-ФЗ нет требования к периодическому переподтверждению. GDPR не применим (российская аудитория, не EU-резиденты). Раздражение возвращающегося пользователя > пользы.

**Последствия.** Согласие действует до явного отзыва пользователем через подвал. Этого достаточно для российской аудитории. Если в будущем сайт начнёт ориентироваться на EU — пересмотр (отдельно от смены подхода к английской локали).

### 2.5. Светлая тема по умолчанию при недоступности системной настройки

**Контекст.** При первой загрузке без сохранённого выбора в `localStorage` и без доступной `prefers-color-scheme` нужно выбрать дефолтную тему.

**Решение.** Светлая тема как умолчание.

**Альтернативы.** Тёмная тема как умолчание — авторская эстетика, минимизирует out-of-flash для технической аудитории с тёмной системной темой. Но: случай «системная настройка недоступна» возникает только на очень старых браузерах, не входящих в требования совместимости (последние 2 версии современных браузеров — `prefers-color-scheme` доступна всегда). Светлая тема — стандартное умолчание, не удивляет.

**Последствия.** Никаких практических — fallback на этот случай почти никогда не сработает в реальных условиях.

## 3. Безопасность и антиспам

### 3.1. Plain text для `shared/failed/`, без шифрования

**Контекст.** При недоступности SMTP обращение сохраняется на диск для последующего ручного разбора. Можно хранить как plain text JSON или зашифровать симметричным AES-256-GCM с ключом из `.env`.

**Решение.** Plain text JSON с правами `0600`, владелец — учётная запись сервиса.

**Альтернативы.** Шифрование AES-256-GCM с ключом из `.env`. Модель угроз для этого шифрования: при компрометации сервера (доступ к диску + к `.env`) защита не работает. Шифрование снижает риск в узких сценариях — изолированная утечка только каталога `failed/` (ошибочные права на бэкап), кража VM-диска или snapshot без доступа к `.env`. Это реальные, но крайне редкие сценарии. Цена шифрования: код шифрования/расшифровки, скрипт `decrypt-failed.sh`, процедура ротации ключа (с особенностью «прямая замена ключа делает существующие файлы недоступными»), упоминание в политике ПДн и реестре обработки.

**Триггер пересмотра.** Появление повышенных требований к защите секретов на диске (сертификация, аудит безопасности) или практический инцидент с утечкой бэкапа.

**Последствия.** Защита — стандартные права UNIX (`0600`, отдельный пользователь). Это совпадает с защитой самого `.env`, которая в любом случае нужна и не дублируется шифрованием. Модель угроз: при компрометации сервера всё уже потеряно, шифрование `failed/` это не меняет.

### 3.2. Honeypot + nginx rate-limit без time-to-submit

**Контекст.** Контактная форма — публичный endpoint, цель для спам-ботов. Возможные уровни защиты: honeypot, time-to-submit, rate-limit, криптографические токены, CAPTCHA.

**Решение.** Два уровня: honeypot-поле и nginx rate-limit (10 r/m + burst 5 по IP).

**Альтернативы.** Time-to-submit как третий уровень — слабый клиентский сигнал, передаётся клиентом и не защищён от подделки (бот пришлёт любое число). Полезен только в комбинации, самостоятельной защитой не является. Криптографически защищённый вариант (подписанный nonce при рендере страницы) — больше кода, но всё ещё обходится при достаточной мотивации спамера. CAPTCHA — Google reCAPTCHA добавляет трансграничную передачу IP в Google, hCaptcha — то же в Cloudflare, обе портят UX.

**Триггер пересмотра.** Появление спама, систематически проходящего через honeypot и rate-limit. Сначала — попробовать криптографически подписанный time-to-submit как самый дешёвый вариант. CAPTCHA — последний resort.

**Последствия.** Honeypot ловит простых ботов. Rate-limit защищает от любых атак на уровне IP. Цена ложного срабатывания honeypot — низкая (только при сознательном вмешательстве пользователя в DOM, чего обычные люди не делают).

### 3.3. CSP с `'unsafe-inline'` в `style-src`

**Контекст.** Строгая CSP запрещает inline-стили. Astro в ряде сценариев инжектит inline `<style>`-теги в HTML (например, для критичного CSS первого экрана).

**Решение.** Допустить `'unsafe-inline'` в `style-src` на старте.

**Альтернативы.** Использовать `'unsafe-hashes'` с хешами всех inline-стилей — Astro 6 умеет генерировать хеши через `security.csp.directives`, но не во всех сценариях; требует ручной проверки и поддержки. nonce-based CSP — несовместим со статическим хостингом (nonce должен генерироваться на каждый запрос). Полностью убрать inline-стили — требует отказа от части оптимизаций Astro.

**Триггер пересмотра.** Когда Astro 6 даст стабильный механизм генерации хешей для всех inline-стилей; или при подаче в bug-bounty / security audit, где `'unsafe-inline'` будет помечен как issue.

**Последствия.** csp-evaluator.withgoogle.com покажет это как «High Risk». Принимается сознательно. На практике риск XSS через inline-стили на сайте, где нет user-generated HTML, — минимальный.

### 3.4. Без HSTS preload list и `/.well-known/security.txt`

**Контекст.** Дополнительные меры безопасности: подача в HSTS preload list (браузеры заранее знают о принудительном HTTPS), `/.well-known/security.txt` (контакт для сообщений об уязвимостях).

**Решение.** Ни то, ни другое не реализуется.

**Альтернативы.** HSTS preload — экономит первый редирект на http://, но: подача через hstspreload.org квази-необратима (удаление занимает месяцы), бенефит для персонального сайта без bug-bounty — невидим (пользователи приходят по https-ссылкам из поиска и закладок). `security.txt` — стандарт RFC 9116 для крупных публичных сервисов, к которым приходят security-researchers. Персональный сайт-визитка не привлекает такое внимание.

**Триггер пересмотра.** HSTS preload — после 6+ месяцев стабильной работы HTTPS без mixed content, если рассматривается как зрелое production-решение. `security.txt` — если сайт начнёт выполнять функцию связанную с обработкой ПДн или платежей, выходящую за рамки контактной формы.

**Последствия.** Заголовок `Strict-Transport-Security` отправляется без директивы `preload`. Файл `security.txt` отсутствует.

### 3.5. Различие 200 vs 204 — аналитика, не защита

**Контекст.** При срабатывании honeypot или иной защиты от спама обработчик возвращает 204 No Content. Документ описывает это как «защиту» в нескольких местах.

**Решение.** 204 — это сигнал для аналитики (цель `form_submit_success` в Метрике не триггерится). Это **не защита от ботов** в прямом смысле: бот, читающий HTTP-статус, легко обнаружит, что попал в honeypot, и адаптирует поведение.

**Обоснование.** Защитную функцию выполняют honeypot (отбрасывание запроса) и rate-limit (отбрасывание запроса). 204 — это просто способ не загрязнить конверсионные метрики ботами, не обнаруживая для бота, что он раскрыт (бот видит «успех», аналитика — фильтрует).

**Последствия.** Зрелый адаптивный спам легко обходит honeypot+204. Если такой спам появится — поднимать уровень защиты (см. §3.2 триггер пересмотра).

## 4. Эксплуатация

### 4.1. Без резервного BCC-ящика у независимого провайдера

**Контекст.** Email-копия обращения у автора — основной юридический след. Можно настроить автоматический BCC на резервный ящик у другого провайдера (страховка от потери доступа к основному провайдеру или случайного удаления).

**Решение.** BCC не настраивается.

**Альтернативы.** BCC на ProtonMail/другого провайдера — добавляет: ещё одного получателя (упоминание в политике), потенциальную трансграничную передачу (если ProtonMail), второй секрет в `.env`, проверку доставки в обе копии в smoke-test.

**Триггер пересмотра.** Реальный инцидент с потерей доступа к основному почтовому провайдеру или массовое удаление.

**Последствия.** Юридический след — только в основном почтовом ящике. Страховка — настроить IMAP-архивирование к себе на диск (бесплатно, без второго провайдера, без отдельной трансграничной передачи). Это операционная задача автора, не часть ТЗ.

### 4.2. Сокращённый прогрев DMARC

**Контекст.** Стандартная процедура перехода DMARC от `p=none` к `p=reject` для крупных рассылок занимает 9 месяцев (none → quarantine pct=10 → 25 → 50 → 100 → reject). Сайт отправляет редкие письма с одной формы через relay-провайдера.

**Решение.** Сокращённая процедура: старт на `p=none`, через 30 дней при чистых отчётах — `p=quarantine; pct=100`, через ещё 60 дней — `p=reject`. Всего ~3 месяца вместо 9.

**Альтернативы.** Полный 9-месячный прогрев — рассчитан на риск массового отказа в доставке при переключении в `reject`. Для редких писем через relay-провайдера, где SPF и DKIM настроены корректно, риск минимальный — DMARC-отчёты быстро покажут любую проблему.

**Триггер пересмотра.** Если DMARC-отчёты покажут ложноположительные карантины — продлить шаг или вернуться к `p=quarantine` пока проблема не будет устранена.

**Последствия.** Быстрее доходим до строгой политики (защита от спуфинга домена). При проблемах легко откатиться — `quarantine` всё ещё лучше, чем `reject` при ложных срабатываниях.

### 4.3. Smoke-test без реальной отправки на `/api/contact`

**Контекст.** После каждого деплоя нужен быстрый smoke-test работоспособности. Один из вариантов — реальная тестовая отправка формы с доставкой в служебный канал.

**Решение.** Smoke-test проверяет: главная отдаёт 200 и содержит ключевой контент, `/en/` отдаёт 200, GET-запрос на `/api/contact` возвращает 405, несуществующий URL возвращает 404. **Реальная отправка POST не выполняется.**

**Альтернативы.** Реальная отправка тестового валидного запроса — даёт уверенность в полном цикле, но создаёт спам в почте автора (несколько деплоев в неделю × тестовые письма). Альтернатива: флаг `dry_run` в payload, который проверяет валидацию и обработку, но не вызывает SMTP. Это допустимо, но добавляет код пути dry-run в SSR-обработчик, увеличивает покрытие тестами этой ветки.

**Триггер пересмотра.** Если фактически обнаружится регрессия SMTP-доставки, не пойманная smoke-тестом.

**Последствия.** Тестирование реальной доставки — ручное, перед каждой существенной публичной версией (часть приёмки, не CI).

### 4.4. Lighthouse-отчёты не архивируются в репозиторий

**Контекст.** Lighthouse запускается перед каждой существенной публичной версией. Результаты можно сохранять в `docs/lighthouse/{date}/` для diff между релизами.

**Решение.** Архивирование не выполняется. Lighthouse запускается, результаты просматриваются автором, действия — точечные.

**Альтернативы.** Архивирование JSON-отчётов в git — даёт исторический trend, помогает обнаружить регрессии. Но: при темпе релизов «несколько раз в месяц» и метриках, стабильно соответствующих порогам приёмки, ценность исторических данных низка. CI Lighthouse (например, через Lighthouse CI Action) — удлинило бы пайплайн на 2-3 минуты ради метрики, которая редко меняется драматически между релизами.

**Триггер пересмотра.** Систематические регрессии Performance, замеченные только постфактум.

**Последствия.** Раздел `docs/lighthouse/` в репозитории отсутствует. Ответственность за качество — ручной аудит перед релизом.

### 4.5. Внешний uptime-monitoring с первого дня

**Контекст.** Автоматическое обнаружение недоступности сайта — отдельная задача от обнаружения ошибок приложения. Можно отложить до первого инцидента или поставить с первого дня.

**Решение.** Бесплатный внешний uptime-monitoring (UptimeRobot или Healthchecks.io) — с первого дня. Проверка `GET /` каждые 5 минут, уведомление на резервный email при недоступности.

**Альтернативы.** Отложить мониторинг до первого инцидента — сэкономить 10 минут на настройке. Но: первый инцидент с потерянной заявкой может произойти раньше, чем автор успеет настроить мониторинг. Стоимость настройки настолько низкая, что откладывать нет смысла. Внутренний healthcheck (cron-скрипт на VPS, проверяющий собственные процессы) — не сработает, если упадёт сам VPS.

**Последствия.** Базовая страховка от «упало и я не знаю». Не покрывает: ошибки приложения (SSR упал, но nginx отдаёт статику — `GET /` возвращает 200), потерянные заявки (SMTP молча не работает). Эти случаи — отложенное расширение (см. §13.5 в [`spec.md`](./spec.md)).

### 4.6. Обязательство еженедельной проверки `shared/failed/`

**Контекст.** При недоступности SMTP заявка падает в `shared/failed/`. Автоматическое уведомление автору не настроено (см. §13.7 в [`spec.md`](./spec.md)). Файлы автоматически удаляются через 90 дней.

**Решение.** Автор берёт на себя **обязательство проверять `shared/failed/` не реже раза в неделю** через личный календарь как повторяющуюся задачу. Процедура — в [`runbook.md`](./runbook.md).

**Обоснование.** При обещании ответа в течение суток и автоматическом удалении через 90 дней — окно потери заявки реально. Еженедельная проверка ограничивает максимальную задержку ответа значением «7 дней + время реакции автора», что для уважительных причин (отпуск, болезнь) приемлемо.

**Триггер пересмотра.** Любая реальная потеря заявки → активация §13.7 (автоматическое уведомление при появлении файла) сразу, без обсуждения.

**Последствия.** Это организационное обязательство, а не техническое. Включено в раздел 12.9 приёмки как пункт «процедура включена в личный календарь».

## 5. Аналитика

### 5.1. Яндекс Метрика как единственный инструмент аналитики

**Контекст.** Доступные инструменты: Яндекс Метрика, Google Analytics, self-hosted Plausible / Umami, иные.

**Решение.** Яндекс Метрика. Альтернативы не подключаются на старте.

**Обоснование.** Основная аудитория — российский рынок, для которого Метрика даёт корректные данные и интегрируется с Яндекс Вебмастером. Google Analytics — трансграничная передача в Google LLC, требующая отдельного согласия и уведомления РКН. Plausible / Umami self-hosted — без cookies (баннер можно убрать), но без интеграции с Вебмастером и с меньшей точностью на российской аудитории.

**Триггер пересмотра.** Существенный англоязычный трафик, для которого Метрика менее информативна. Тогда — добавление self-hosted Plausible или Umami как параллельного инструмента (без cookies, без баннера для англоязычной аудитории, см. §13 в [`spec.md`](./spec.md) в исходной версии).

**Последствия.** Один инструмент, одна точка интеграции, один баннер согласия. Cookies от Метрики — единственная категория, требующая согласия.

### 5.2. Webvisor и карта кликов выключены

**Контекст.** Яндекс Метрика опционально предоставляет Webvisor (запись сессий) и карту кликов.

**Решение.** Не подключаются.

**Обоснование.** Webvisor потенциально захватывает содержимое полей формы при наборе текста, что нарушает принцип минимизации обработки ПДн (форма содержит имя, email, детали проекта — это персональные данные). Webvisor исторически требует ослабления CSP до `'unsafe-eval'`, что значимо снижает защиту от XSS.

**Триггер пересмотра.** Сознательное решение углубить UX-аналитику, с пересмотром CSP, согласия пользователя на запись и описания в политике ПДн.

**Последствия.** CSP не содержит `'unsafe-eval'`. Раздел политики о Метрике не упоминает запись сессий. Если когда-то понадобится — это отдельный полный пересмотр с юридической частью.

## 6. Тестирование

### 6.1. Без E2E-тестов и visual regression

**Контекст.** Современный стек тестирования включает unit, integration, E2E (Playwright/Cypress), visual regression (Percy, Chromatic). Что минимально достаточно?

**Решение.** Unit-тесты только на критичные утилиты, smoke-test после деплоя, ручная регрессия по чек-листу приёмки перед существенными версиями. E2E и visual regression не реализуются.

**Обоснование.** E2E через Playwright/Cypress удваивает время на каждый PR без пропорциональной пользы при текущем темпе изменений и сложности интерактива. Visual regression при отсутствии команды дизайнеров и редких изменений вёрстки — overhead.

**Триггер пересмотра.** Добавление сложного интерактива (фильтрация, состояние в URL, многошаговые формы) — кандидат на E2E. Появление систематических визуальных регрессий после рефакторингов — кандидат на visual regression.

**Последствия.** Ответственность за регрессии — на ручной чек-лист приёмки. Это работает при одном разработчике и редких изменениях; при расширении команды — пересмотр.

### 6.2. Unit-тесты только на критичные утилиты

**Контекст.** Возможно покрытие тестами от 0% до 100%. Что выбрать?

**Решение.** Покрываются только функции, цена ошибки в которых — потеря заявок или нарушение безопасности: HTML-экранирование, CRLF-санитизация для заголовков письма, расчёт времени чтения, извлечение IP клиента, валидация полей формы.

**Обоснование.** Критичность функции не коррелирует с её размером. Маленькая функция санитизации CRLF — single point of failure для email header injection. Большие функции рендеринга страниц — покрыты Astro и тестируются визуально.

**Последствия.** CI быстрый. Регрессии в визуальных элементах — на ручную приёмку.

## 7. SEO и контент

### 7.1. `x-default` указывает на русскую версию

**Контекст.** Тег `<link rel="alternate" hreflang="x-default">` указывает поисковикам, какой вариант показывать при отсутствии явного языкового совпадения с пользователем.

**Решение.** `x-default` → русская версия.

**Обоснование.** Основной рынок — русскоязычный. Англоязычная аудитория обрабатывается опциональной локалью с частичным контентом. Если поисковик не знает язык посетителя — лучше показать наполненную RU-страницу, чем потенциально пустую EN.

**Триггер пересмотра.** При наполнении EN-локали и росте англоязычного трафика > 25% — пересмотр в пользу EN (см. §13.3 в [`spec.md`](./spec.md)).

**Последствия.** Английская локаль теряет небольшой долю трафика с неопределённым языком пользователя. На старте это незначимо.

### 7.2. Schema.org image — один вариант 1200×630

**Контекст.** Google Search Central рекомендует три варианта изображения для постов (1:1, 4:3, 16:9) для оптимизации rich-результатов.

**Решение.** Один вариант 1200×630 (16:9, минимум для OG).

**Обоснование.** Три варианта — для блогов с агрессивной SEO-стратегией и большим трафиком из Discover. Для блога с темпом 1-2 поста в месяц прирост от трёх вариантов незаметен. Один вариант 1200×630 проходит Google Rich Results Test без warning-ов.

**Последствия.** Каждый пост требует одной картинки нужного размера. Меньше работы при подготовке контента.

### 7.3. OG-картинки через универсальный шаблон, без per-page satori-генерации

**Контекст.** Open Graph картинки для соцсетей. Варианты: один статический `og-default.png` на весь сайт, статический шаблон с подставляемым заголовком, динамическая генерация через satori для каждой страницы.

**Решение.** Универсальный шаблон с подставляемым заголовком на сборке. Резерв — статический `og-default.png` для страниц без заголовка.

**Обоснование.** Полностью статический OG — слабая дифференциация в соцсетях. Полная per-page satori-генерация — добавляет сложную зависимость, увеличивает время сборки, требует поддержки шрифтов с кириллическим набором. Шаблон с подставленным заголовком — золотая середина.

**Последствия.** Все страницы (главная, посты, проекты, решения) имеют визуально похожие OG-картинки с фирменным фоном и индивидуальным заголовком.
