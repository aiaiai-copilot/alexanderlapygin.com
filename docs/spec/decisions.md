# Журнал решений (decisions.md)

Документ фиксирует **архитектурные и продуктовые решения** проекта alexanderlapygin.com с обоснованиями и отвергнутыми альтернативами. Это сопровождение к [`spec.md`](./spec.md) — само ТЗ описывает *что* система делает, этот документ — *почему именно так*.

Формат каждой записи: контекст → решение → отвергнутые альтернативы → последствия.

## 1. Архитектурные решения

### 1.1. Полностью статический сайт без серверной части

**Контекст.** Сайт — read-only витрина с одной интерактивной точкой (форма обращения). Изначально форма обращения проектировалась с серверным эндпоинтом для отправки email через SMTP — это вынуждало держать SSR-процесс (Node.js на VPS), nginx с rate-limit, systemd, secrets-managment, ротацию логов, резервный канал `failed/` с еженедельной проверкой и инфраструктурный код экранирования/санитизации. После пересмотра канала доставки (см. §1.2) серверная часть стала ненужной.

**Решение.** Astro в режиме `output: 'static'`. Все страницы пререндерятся на сборке. Серверных эндпоинтов нет. Деплой — публикация `dist/` на статический хостинг.

**Альтернативы.**

- *Statics + serverless function для формы* (Cloudflare Workers / Yandex Cloud Functions): даёт возможность сохранять структурированный приём, серверный anti-spam, и почту/Telegram-бот доставку. Но сохраняет всю юридическую/инфраструктурную поверхность серверной части (трансгран при зарубежной функции, секреты, серверная валидация и тесты на неё, мониторинг функции). Отвергнуто как преждевременная сложность для трафика «несколько обращений в месяц».
- *SSG + сторонний form-сервис* (Formspree, Web3Forms, Static Forms): добавляет трансграничную передачу всех полей формы внешнему получателю (всегда зарубежный), требует уведомления РКН по ст. 12 и отражения в политике. Эквивалентно по юридике зарубежной serverless-функции, без выигрыша.
- *Полный SSR* (Node + nginx + systemd): уже спроектирован и отвергнут в пользу deeplink (см. §1.2). Ставит Node-процесс в критический путь, добавляет surface area, требует graceful restart или blue-green; для трафика проекта — оверкилл.

**Триггер пересмотра.** §13.5 в [`spec.md`](./spec.md) — возврат к backend-доставке при систематической невозможности принимать обращения через deeplink/mailto.

**Последствия.** Минимальная инфраструктура: статический хостинг + DNS. Нет процессов для мониторинга, нет секретов на сервере, нет ротации логов, нет резервного канала `failed/`. Юридическая поверхность сужается: оператор сайта не передаёт ПДн третьим лицам через форму. Цена — потеря серверного контроля (anti-spam, структурированный приём, гарантия доставки), см. §2.1.

### 1.2. Канал доставки обращений — Telegram deeplink

**Контекст.** Возможные каналы доставки обращений от посетителя автору: email через SMTP, Telegram через Bot API (`sendMessage` от имени сервиса), Telegram через deeplink (`https://t.me/<username>?text=...`, отправка инициируется самим посетителем из его клиента), иные мессенджеры. Каждый канал имеет разный юридический и инфраструктурный профиль.

**Решение.** Telegram deeplink как основной канал, `mailto:`-ссылка как параллельный fallback (в подвале и в правой колонке `/contact`).

**Альтернативы.**

- *Email через SMTP*: классический канал, юридически чистый при российском relay-провайдере. Цена: SMTP-провайдер, прогрев SPF/DKIM/DMARC (~3 месяца), `mail-tester.com`-аудит, проблема deliverability (письма в спам, отсутствие гарантий доставки в современную почту), необходимость serverless или SSR-функции для отправки. По факту канал отвергнут после совокупной оценки сложности и качества доставки.
- *Telegram Bot API через backend*: оперативное уведомление, но backend инициирует трансграничную передачу в Telegram Messenger Inc. (вне РФ). Это активирует ст. 12 152-ФЗ: уведомление в РКН до начала эксплуатации, отдельное согласие субъекта, отражение в политике конфиденциальности, отдельный путь экранирования (MarkdownV2/HTML), отдельная процедура удаления сообщений при запросе субъекта.
- *Telegram deeplink* (выбранный вариант): передачу инициирует посетитель из своего клиента, оператор сайта в передаче не участвует — юридическая позиция симметрична `mailto:`. Backend не нужен, что разблокировало §1.1.

**Обоснование выбора deeplink.**

- *Юридика.* Оператор не передаёт ПДн в зарубежного получателя (Telegram) — это делает сам субъект. Уведомление по ст. 12 152-ФЗ через этот канал не требуется. Политика конфиденциальности упрощается.
- *Инфраструктура.* Сайт остаётся полностью статичным, исчезают serverless-функция/VPS, secrets, server-side anti-spam, серверная валидация, mail-tester и DNS-прогрев.
- *Симметрия с `mailto:`*. Поведение по сути такое же: открывается клиент посетителя с готовым черновиком, отправляет посетитель сам. Юридический режим тот же.
- *UX.* Для пользователей с Telegram (а это типичная аудитория автора — техлиды, малый/средний российский бизнес) канал удобный и быстрый. Для остальных — `mailto:`-fallback.

**Цена решения.**

- Server-side anti-spam (honeypot + nginx rate-limit) теряется — но и поверхность атаки исчезает (нечего спамить, нет HTTP-эндпоинта, принимающего тело).
- Структурированный ввод деградирует до префилла в свободном тексте: `<select>` ограничивает значение на клиенте, но серверной гарантии нет, посетитель может отредактировать черновик в Telegram перед отправкой.
- Префилл ограничен длиной URL (Telegram держит ~4 КБ); лимит textarea `details` снижается с 5000 до 2000 символов, при превышении — обрезка с маркером.
- Аналитическое различие «успешно отправлено» / «отброшено как спам» — отсутствует; событие `form_submit_telegram` триггерится по клику на кнопку, не по факту отправки в Telegram.
- Отсутствие серверного канала «сохранить в `failed/` при недоставке»: если посетитель не нажал «Отправить» в Telegram — обращение потеряно. Это симметрично `mailto:` (письмо может попасть в спам или не быть отправлено).

**Триггер пересмотра.** §13.5 в [`spec.md`](./spec.md): возврат к backend-доставке (SMTP или Telegram Bot API) при систематической невозможности принимать обращения через deeplink/mailto или появлении требований CRM-интеграции.

**Последствия.** Деплой = публикация статики; никаких серверных секретов; политика конфиденциальности упрощается; уведомление в РКН по ст. 12 не требуется при российском хостинге.

### 1.3. certbot и сертификаты — не нужны

**Контекст.** В исходной архитектуре (SSR на VPS) HTTPS-сертификат выпускался через certbot с автопродлением.

**Решение.** Использование штатного автоматического HTTPS на стороне статического хостинга. certbot, его конфигурация и DNS-плагины не нужны.

**Обоснование.** Все рассматриваемые статические хостинги (Cloudflare Pages, GitHub Pages, Yandex Cloud Object Storage в режиме сайта, Selectel, Beget) штатно выпускают и продлевают сертификаты для кастомных доменов. Это снимает класс проблем (выбор HTTP-01 vs DNS-01, community-плагины DNS-провайдеров, ротация, мониторинг истечения).

**Триггер пересмотра.** Только если выбранный хостинг откажет в HTTPS (что для современных провайдеров — гипотетический сценарий).

**Последствия.** Раздел про certbot из runbook удаляется. При смене хостинга — заново настраивается через UI/CLI нового провайдера.

### 1.4. Английская локаль с первого дня инфраструктурно

**Контекст.** Основная аудитория — русскоязычная. Англоязычная — вторичная, по факту неподтверждённая. Возможны два пути: построить двуязычную инфраструктуру сразу или начать только с RU.

**Решение.** Двуязычная инфраструктура с первого дня (маршруты `/en/`, словари UI, переключатели, hreflang, x-default). Контент в EN — частичный, наполняется по триггеру (§13.3 в [`spec.md`](./spec.md)).

**Альтернативы.** Только RU на старте, EN добавляется позже целиком. Это упростило бы инфраструктуру (один набор маршрутов, нет переключателей, нет hreflang/x-default, одна 404, одна ценовая сетка), но при появлении триггера потребовало бы редизайна URL-структуры, что болезненно для SEO существующих RU-страниц.

**Последствия.** Постоянный overhead двуязычности даже при пустой EN-локали. Зато добавление EN-контента — точечная редактура Markdown-файлов, без структурных изменений. При выборе сценария «никогда не понадобится EN» — overhead был построен впустую; вероятность этого сценария оценивается автором как невысокая, но реальная. Это сознательный риск.

### 1.5. Выбор хостинг-провайдера — отдельное решение

**Контекст.** На каком статическом хостинге разворачивается проект. Решение влияет на: применимость ст. 12 152-ФЗ (трансграничная передача при зарубежном хостинге — HTTP-логи и раздача файлов за пределами РФ), стоимость, скорость CDN, доступность кастомных HTTP-заголовков (требование, см. §9.1 в [`spec.md`](./spec.md)), удобство интеграции с GitHub Actions.

**Решение.** Конкретный провайдер **не зафиксирован в** [`spec.md`](./spec.md). На прототипе используется Cloudflare Pages. Финальный выбор делается до публикации публичной версии и фиксируется в [`runbook.md`](./runbook.md).

**Кандидаты без трансграничной передачи (российские):** Yandex Cloud Object Storage (как статический сайт), Selectel Object Storage, Beget статический хостинг, Cloud.ru, Timeweb.

**Кандидаты с трансграничной передачей:** Cloudflare Pages (используется на прототипе), GitHub Pages, Netlify, Vercel, AWS S3 + CloudFront.

**Последствия зарубежного выбора:**

- Активируется раздел 10.4 spec.md в части хостинга.
- Требуется уведомление Роскомнадзора по ст. 12 152-ФЗ до публичного запуска.
- В политику конфиденциальности добавляется конкретный получатель и страна хостинга.

**Сходство всех провайдеров.** Архитектура одинаковая (статика + CDN + штатный HTTPS + кастомные заголовки). Перенос между провайдерами — собрать `dist/`, опубликовать через CLI нового провайдера, изменить DNS-запись. Время полной миграции — час-два.

## 2. Продуктовые компромиссы

### 2.1. Один канал доставки обращений (Telegram deeplink) + email-fallback

**Контекст.** Возможные каналы доставки обращений автору: email через `mailto:`, Telegram через deeplink, Telegram-бот через backend `sendMessage`, серверная отправка email через SMTP, формы внешних сервисов.

**Решение.** Telegram deeplink — основной, `mailto:` — параллельный fallback. Обе ссылки доступны без JavaScript.

**Альтернативы.** Подробно рассмотрены в §1.2.

**Триггер пересмотра.** См. §13.5 в [`spec.md`](./spec.md).

**Последствия.** На стороне сайта не накапливаются персональные данные — оператор не передаёт ПДн третьим лицам через канал формы. Цена: контроль над поверхностью приёма теряется (нельзя гарантировать структурированный ввод, нельзя ловить ботов на серверной стороне). Цена приемлема, так как server-side спам-поверхности больше нет.

### 2.2. Прямые контакты только в подвале и на `/contact`

**Контекст.** Контактные ссылки (email, Telegram, GitHub) могут размещаться в шапке, в тексте `/about`, в подвале, на специальной странице `/contact`. Чем больше точек контакта — тем выше шанс «прямого обращения», но тем менее типизирована воронка.

**Решение.** Контакты — только в двух местах: сквозной подвал (с иконками email, Telegram, GitHub) и страница `/contact` (форма + прямые ссылки email/Telegram). В шапке контактных иконок нет. В тексте `/about` нет `mailto:`-ссылок и аналогов.

**Альтернативы.** Прямые `mailto:`/`t.me`-ссылки в тексте `/about` — повышают конверсию для тех, кто уже готов написать, но снижают долю обращений через форму с префиллом, которая даёт автору минимальный полезный контекст до начала диалога (тип проекта, бюджет, детали). При прямой ссылке — переписка часто начинается с «привет, можете сделать X?» без подробностей.

**Последствия.** Воронка направляет посетителей в форму, при этом не блокирует прямую связь (контакты видны в подвале и на `/contact`). Изменение контактов — точечное в одном месте, без правок текста разделов.

### 2.3. Mobile-first как основной профиль качества

**Контекст.** Качество отображения проверяется на нескольких диапазонах ширины. Можно оптимизировать в первую очередь под десктоп или под мобильные.

**Решение.** Mobile-first. Lighthouse mobile — основной профиль приёмки. Десктоп — производный.

**Обоснование.** Google индексирует сайты в mobile-режиме с 2019 года. Mobile-профиль Lighthouse существенно строже десктопного (эмулируется класс CPU и канал). Значительная доля целевой аудитории приходит с мобильных устройств. Если страница работает хорошо на mobile — на desktop она почти гарантированно работает отлично; обратное неверно.

**Последствия.** Дизайн-решения принимаются от узких экранов вверх. На странице поста допускается чуть более низкий Performance ≥ 75 vs типовые ≥ 80 — сознательный компромисс качества и функциональности (ToC, подсветка синтаксиса, OG-генерация).

### 2.4. Cookie-баннер без 12-месячного истечения согласия

**Контекст.** Cookie-баннер при первом посещении сохраняет факт согласия в `localStorage`. Можно автоматически показывать его повторно по истечении срока (например, 12 месяцев).

**Решение.** Однократный показ баннера. Без автоматического повторного показа. Пользователь может пересмотреть выбор через ссылку «Настройки cookie» в подвале.

**Альтернативы.** Повторный показ через 12 месяцев — даёт «свежее согласие», но в 152-ФЗ нет требования к периодическому переподтверждению. GDPR не применим (российская аудитория, не EU-резиденты). Раздражение возвращающегося пользователя > пользы.

**Последствия.** Согласие действует до явного отзыва пользователем через подвал. Этого достаточно для российской аудитории. Если в будущем сайт начнёт ориентироваться на EU — пересмотр (отдельно от смены подхода к английской локали).

### 2.5. Светлая тема по умолчанию при недоступности системной настройки

**Контекст.** При первой загрузке без сохранённого выбора в `localStorage` и без доступной `prefers-color-scheme` нужно выбрать дефолтную тему.

**Решение.** Светлая тема как умолчание.

**Альтернативы.** Тёмная тема как умолчание — авторская эстетика, минимизирует out-of-flash для технической аудитории с тёмной системной темой. Но: случай «системная настройка недоступна» возникает только на очень старых браузерах, не входящих в требования совместимости (последние 2 версии современных браузеров — `prefers-color-scheme` доступна всегда). Светлая тема — стандартное умолчание, не удивляет.

**Последствия.** Никаких практических — fallback на этот случай почти никогда не сработает в реальных условиях.

### 2.6. Email — необязательное поле в форме

**Контекст.** При канале доставки через Telegram deeplink основной обратный канал автору — Telegram-аккаунт отправителя (виден в Telegram при получении сообщения). Email становится дублирующей информацией.

**Решение.** Поле Email в форме — необязательное. Обязательные поля — только Имя и Детали проекта.

**Альтернативы.** Сохранить Email обязательным — даёт автору альтернативный канал обратной связи (на случай, если ответ через Telegram заблокирован настройками отправителя или не доходит). Цена — ниже конверсия (одно лишнее поле для заполнения), при том что Telegram-аккаунт отправителя автору всё равно виден.

**Триггер пересмотра.** Если систематически встречаются ситуации, когда автор не может связаться с отправителем через Telegram (приватные настройки, заблокированный ответ) — вернуть Email в обязательные поля или добавить отдельную просьбу указать email прямо в шаблоне префилла.

**Последствия.** Меньше трения в форме, конверсия выше. На стороне Telegram-сообщения поле Email отображается только при заполнении.

## 3. Безопасность и антиспам

### 3.1. Нет серверного anti-spam — нет серверной поверхности

**Контекст.** Исходная архитектура с HTTP-эндпоинтом `/api/contact` требовала двухуровневой защиты от спама (honeypot + nginx rate-limit). При переходе на deeplink эндпоинта нет.

**Решение.** Серверный anti-spam не реализуется (некуда применять). Клиентская валидация полей сохраняется только для UX (не для защиты).

**Обоснование.** Поверхность для автоматизированного спама исчезла: чтобы «отправить» через deeplink, бот должен открыть Telegram-клиент, нажать «Отправить» вручную в чужом клиенте — это естественный фильтр. Единственный реалистичный канал спама — массовая отправка человеком с реальных Telegram-аккаунтов, что не отличается от обычной нежелательной переписки и фильтруется штатными средствами Telegram (блокировка пользователя).

**Триггер пересмотра.** При возврате к backend-каналу (§13.5 в [`spec.md`](./spec.md)) — заново ставится honeypot + rate-limit как минимум.

**Последствия.** Раздел spec'а про серверную защиту удалён. Unit-тесты на HTML-экранирование и CRLF-санитизацию (нужные для email) — удалены, так как нет SMTP-канала.

### 3.2. CSP с `'unsafe-inline'` в `style-src`

**Контекст.** Строгая CSP запрещает inline-стили. Astro в ряде сценариев инжектит inline `<style>`-теги в HTML (например, для критичного CSS первого экрана).

**Решение.** Допустить `'unsafe-inline'` в `style-src` на старте.

**Альтернативы.** Использовать `'unsafe-hashes'` с хешами всех inline-стилей — Astro 6 умеет генерировать хеши, но не во всех сценариях; требует ручной проверки и поддержки. nonce-based CSP — несовместим со статическим хостингом (nonce должен генерироваться на каждый запрос). Полностью убрать inline-стили — требует отказа от части оптимизаций Astro.

**Триггер пересмотра.** Когда Astro 6 даст стабильный механизм генерации хешей для всех inline-стилей; или при подаче в bug-bounty / security audit, где `'unsafe-inline'` будет помечен как issue.

**Последствия.** csp-evaluator.withgoogle.com покажет это как «High Risk». Принимается сознательно. На практике риск XSS через inline-стили на сайте, где нет user-generated HTML, — минимальный.

### 3.3. Без HSTS preload list и `/.well-known/security.txt`

**Контекст.** Дополнительные меры безопасности: подача в HSTS preload list (браузеры заранее знают о принудительном HTTPS), `/.well-known/security.txt` (контакт для сообщений об уязвимостях).

**Решение.** Ни то, ни другое не реализуется.

**Альтернативы.** HSTS preload — экономит первый редирект на http://, но: подача через hstspreload.org квази-необратима (удаление занимает месяцы), бенефит для персонального сайта без bug-bounty — невидим. `security.txt` — стандарт RFC 9116 для крупных публичных сервисов, к которым приходят security-researchers. Персональный сайт-визитка не привлекает такое внимание.

**Триггер пересмотра.** HSTS preload — после 6+ месяцев стабильной работы HTTPS без mixed content, если рассматривается как зрелое production-решение. `security.txt` — если сайт начнёт выполнять функцию, связанную с обработкой ПДн или платежей, выходящую за рамки контактной формы.

**Последствия.** Заголовок `Strict-Transport-Security` отправляется без директивы `preload`. Файл `security.txt` отсутствует.

### 3.4. URL-кодирование как единственная защита подстановки в Telegram-URL

**Контекст.** При формировании Telegram deeplink значения полей формы подставляются в query-string `?text=...`. Без должного кодирования специальные символы (`&`, `?`, `#`, перевод строки) ломают URL.

**Решение.** Все значения проходят `encodeURIComponent` перед конкатенацией. Telegram-username фиксируется в build-time, не зависит от пользовательского ввода.

**Обоснование.** Это стандартный API браузера, корректно покрывающий весь набор символов URL-кодировки. Никакой дополнительной HTML- или MarkdownV2-санитизации не требуется, потому что текст не парсится как разметка — Telegram-клиент показывает его как plain text черновика, который пользователь сам отправит как обычное сообщение.

**Альтернативы.** Своя реализация кодирования — нет смысла. Дополнительная HTML-санитизация — не нужна, потому что текст не попадает в HTML-контекст ни на этапе формирования URL (это URL, не HTML), ни в Telegram (где сообщение пишется как plain text).

**Последствия.** Простая, понятная защита от единственного класса проблем (поломка URL). Нет инфраструктурного кода санитизации, нет тестов на CRLF/HTML-инъекции для серверной стороны (потому что серверной стороны нет).

## 4. Эксплуатация

### 4.1. Smoke-test проверяет только статические маршруты

**Контекст.** После каждого деплоя нужен быстрый smoke-test работоспособности.

**Решение.** Smoke-test проверяет: главная RU отдаёт 200, главная EN отдаёт 200, несуществующий URL возвращает 404. Никаких POST-запросов нет, потому что нет HTTP-эндпоинтов.

**Альтернативы.** Дополнительно дёргать страницу `/contact` и проверять наличие Telegram-deeplink в HTML — переусложнение для текущей фазы, статический рендер уже покрыт типами и валидацией коллекций на сборке.

**Последствия.** Smoke-test быстрый, не зависит ни от каких внешних сервисов (Telegram, SMTP). Регрессии в самой логике формы — на ручной чек-листе приёмки и unit-тестах на формирование префилла.

### 4.2. Lighthouse-отчёты не архивируются в репозиторий

**Контекст.** Lighthouse запускается перед каждой существенной публичной версией. Результаты можно сохранять в `docs/lighthouse/{date}/` для diff между релизами.

**Решение.** Архивирование не выполняется. Lighthouse запускается, результаты просматриваются автором, действия — точечные.

**Альтернативы.** Архивирование JSON-отчётов в git — даёт исторический trend, помогает обнаружить регрессии. Но: при темпе релизов «несколько раз в месяц» и метриках, стабильно соответствующих порогам приёмки, ценность исторических данных низка. CI Lighthouse (например, через Lighthouse CI Action) — удлинило бы пайплайн на 2-3 минуты ради метрики, которая редко меняется драматически между релизами.

**Триггер пересмотра.** Систематические регрессии Performance, замеченные только постфактум.

**Последствия.** Раздел `docs/lighthouse/` в репозитории отсутствует. Ответственность за качество — ручной аудит перед релизом.

### 4.3. Внешний uptime-monitoring с первого дня

**Контекст.** Автоматическое обнаружение недоступности сайта — отдельная задача. Можно отложить до первого инцидента или поставить с первого дня.

**Решение.** Бесплатный внешний uptime-monitoring (UptimeRobot или Healthchecks.io) — с первого дня. Проверка `GET /` каждые 5 минут, уведомление на резервный email при недоступности.

**Альтернативы.** Отложить мониторинг до первого инцидента — сэкономить 10 минут на настройке. Но: первый инцидент с тихой недоступностью может произойти раньше, чем автор успеет настроить мониторинг. Стоимость настройки настолько низкая, что откладывать нет смысла.

**Последствия.** Базовая страховка от «упало и я не знаю». Покрытие исчерпывается доступностью статики; ошибки клиентского JS (сломанное формирование deeplink) — отлавливаются Метрикой (отсутствие срабатываний `form_submit_telegram`) и ручной проверкой при релизе.

## 5. Аналитика

### 5.1. Яндекс Метрика как единственный инструмент аналитики

**Контекст.** Доступные инструменты: Яндекс Метрика, Google Analytics, self-hosted Plausible / Umami, иные.

**Решение.** Яндекс Метрика. Альтернативы не подключаются на старте.

**Обоснование.** Основная аудитория — российский рынок, для которого Метрика даёт корректные данные и интегрируется с Яндекс Вебмастером. Google Analytics — трансграничная передача в Google LLC, требующая отдельного согласия и уведомления РКН. Plausible / Umami self-hosted — без cookies (баннер можно убрать), но без интеграции с Вебмастером и с меньшей точностью на российской аудитории.

**Триггер пересмотра.** Существенный англоязычный трафик, для которого Метрика менее информативна. Тогда — добавление self-hosted Plausible или Umami как параллельного инструмента (без cookies, без баннера для англоязычной аудитории).

**Последствия.** Один инструмент, одна точка интеграции, один баннер согласия. Cookies от Метрики — единственная категория, требующая согласия.

### 5.2. Webvisor и карта кликов выключены

**Контекст.** Яндекс Метрика опционально предоставляет Webvisor (запись сессий) и карту кликов.

**Решение.** Не подключаются.

**Обоснование.** Webvisor потенциально захватывает содержимое полей формы при наборе текста, что нарушает принцип минимизации обработки ПДн. Webvisor исторически требует ослабления CSP до `'unsafe-eval'`, что значимо снижает защиту от XSS.

**Триггер пересмотра.** Сознательное решение углубить UX-аналитику, с пересмотром CSP, согласия пользователя на запись и описания в политике ПДн.

**Последствия.** CSP не содержит `'unsafe-eval'`. Раздел политики о Метрике не упоминает запись сессий. Если когда-то понадобится — это отдельный полный пересмотр с юридической частью.

### 5.3. `form_submit_telegram` — клик, не доставка

**Контекст.** При канале deeplink сайт не знает, нажал ли посетитель «Отправить» в Telegram после открытия черновика. Можно триггерить событие конверсии по клику на кнопку формы (есть факт открытия Telegram), либо вообще не триггерить (есть только факт получения сообщения у автора, вне аналитики).

**Решение.** Триггерить `form_submit_telegram` по клику на активную кнопку «Открыть в Telegram» (т. е. валидация формы прошла, deeplink сформирован, открывается новая вкладка).

**Обоснование.** Это максимум, что может видеть аналитика на стороне сайта. Корреляция «открыл deeplink → отправил сообщение» определяется автором постфактум по соотношению срабатываний цели в Метрике и реальных полученных сообщений.

**Альтернативы.** Не триггерить — потеря промежуточной метрики воронки (сколько посетителей доходит до клика по кнопке). Триггерить по `visibilitychange` после возврата на сайт — ненадёжно (пользователь может закрыть вкладку, перейти куда-то ещё).

**Последствия.** Метрика показывает верхнюю границу «обращений» — реальное число полученных сообщений обычно меньше. Это документируется в политике и в самом decisions.md, чтобы не вводить в заблуждение при чтении отчётов.

## 6. Тестирование

### 6.1. Без E2E-тестов и visual regression

**Контекст.** Современный стек тестирования включает unit, integration, E2E (Playwright/Cypress), visual regression (Percy, Chromatic). Что минимально достаточно?

**Решение.** Unit-тесты только на критичные утилиты, smoke-test после деплоя, ручная регрессия по чек-листу приёмки перед существенными версиями. E2E и visual regression не реализуются.

**Обоснование.** E2E через Playwright/Cypress удваивает время на каждый PR без пропорциональной пользы при текущем темпе изменений и сложности интерактива. Visual regression при отсутствии команды дизайнеров и редких изменений вёрстки — overhead.

**Триггер пересмотра.** Добавление сложного интерактива (фильтрация, состояние в URL, многошаговые формы) — кандидат на E2E. Появление систематических визуальных регрессий после рефакторингов — кандидат на visual regression.

**Последствия.** Ответственность за регрессии — на ручной чек-лист приёмки. Это работает при одном разработчике и редких изменениях; при расширении команды — пересмотр.

### 6.2. Unit-тесты только на критичные утилиты

**Контекст.** Возможно покрытие тестами от 0% до 100%. Что выбрать?

**Решение.** Покрываются только функции, цена ошибки в которых — поломка ключевого UX или некорректная навигация:

- Транслитерация slug (любая регрессия — поломка URL).
- Расчёт времени чтения (виден на каждой странице поста).
- Формирование префилл-текста Telegram-сообщения (порядок полей, переносы строк, `encodeURIComponent`, обрезка при превышении лимита).
- Клиентская валидация полей формы.

**Обоснование.** Критичность функции не коррелирует с её размером. Маленькая функция формирования URL — single point of failure для главной конверсии. Большие функции рендеринга страниц — покрыты Astro и тестируются визуально.

**Что не тестируется и почему.**

- HTML/CRLF-санитизация — не нужна, серверного канала нет.
- Извлечение IP клиента — не нужно, серверного логирования нет.
- Доставка email — не нужна, SMTP нет.

**Последствия.** CI быстрый. Регрессии в визуальных элементах — на ручную приёмку.

## 7. SEO и контент

### 7.1. `x-default` указывает на русскую версию

**Контекст.** Тег `<link rel="alternate" hreflang="x-default">` указывает поисковикам, какой вариант показывать при отсутствии явного языкового совпадения с пользователем.

**Решение.** `x-default` → русская версия.

**Обоснование.** Основной рынок — русскоязычный. Англоязычная аудитория обрабатывается опциональной локалью с частичным контентом. Если поисковик не знает язык посетителя — лучше показать наполненную RU-страницу, чем потенциально пустую EN.

**Триггер пересмотра.** При наполнении EN-локали и росте англоязычного трафика > 25% — пересмотр в пользу EN (см. §13.3 в [`spec.md`](./spec.md)).

**Последствия.** Английская локаль теряет небольшую долю трафика с неопределённым языком пользователя. На старте это незначимо.

### 7.2. Schema.org image — один вариант 1200×630

**Контекст.** Google Search Central рекомендует три варианта изображения для постов (1:1, 4:3, 16:9) для оптимизации rich-результатов.

**Решение.** Один вариант 1200×630 (16:9, минимум для OG).

**Обоснование.** Три варианта — для блогов с агрессивной SEO-стратегией и большим трафиком из Discover. Для блога с темпом 1-2 поста в месяц прирост от трёх вариантов незаметен. Один вариант 1200×630 проходит Google Rich Results Test без warning-ов.

**Последствия.** Каждый пост требует одной картинки нужного размера. Меньше работы при подготовке контента.

### 7.3. OG-картинки через универсальный шаблон, без per-page satori-генерации

**Контекст.** Open Graph картинки для соцсетей. Варианты: один статический `og-default.png` на весь сайт, статический шаблон с подставляемым заголовком, динамическая генерация через satori для каждой страницы.

**Решение.** Универсальный шаблон с подставляемым заголовком на сборке. Резерв — статический `og-default.png` для страниц без заголовка.

**Обоснование.** Полностью статический OG — слабая дифференциация в соцсетях. Полная per-page satori-генерация — добавляет сложную зависимость, увеличивает время сборки, требует поддержки шрифтов с кириллическим набором. Шаблон с подставленным заголовком — золотая середина.

**Последствия.** Все страницы (главная, посты, проекты, решения) имеют визуально похожие OG-картинки с фирменным фоном и индивидуальным заголовком.
